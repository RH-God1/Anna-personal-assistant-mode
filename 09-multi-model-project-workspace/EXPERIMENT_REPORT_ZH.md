# Anna 多模型项目工作区虚拟实验报告

实验日期：2026-06-14

## 实验配置

本实验没有调用真实供应商 API，而是创建了 5 个确定性虚拟模型：

| 模型 | 能力 | 上下文窗口 | 用途 |
| --- | --- | ---: | --- |
| `virtual-architect` | text、code、tools | 4096 | 生成 API 契约 |
| `virtual-vision` | text、vision、audio | 4096 | 分析界面附件 |
| `virtual-synth-small` | text、code | 512 | 汇总两个上游项目 |
| `virtual-text-only` | text | 2048 | 能力不匹配测试 |
| `virtual-offline` | text、code、vision | 8192 | 离线回退测试 |

实验总项目包含 8 个子项目，包括正常工作项目、隔离探针、能力探针和循环依赖探针。

## 协作流程

1. 子项目 `1-接口构建` 使用架构模型生成 `api-contract.json`。
2. 产物初始为私有；依赖项目无法读取。
3. 用户确认发布后，依赖项目可以读取该产物。
4. 子项目 `2-视觉分析` 使用视觉模型处理图片附件元数据，生成 `ui-analysis.md`。
5. 子项目 `3-整合` 使用 512 token 小窗口模型读取两个已发布产物并完成汇总。

小窗口模型的输入预算为 1331 字符。实际使用 1331 字符，两个关键依赖产物均被保留，
5 条低优先级压力记忆被省略，`context_budget.truncated` 正确标记为 `true`。

## 发现与修复

### 1. 模型离线状态判断不完整

原实现只排除 `disabled`，`offline` 或 `unavailable` 模型仍可能被绑定。

修复后模型状态统一转为小写，并通过明确的 `available` 字段参与路由。离线模型会回退
到 `anna-auto`，同时保留具体原因。

### 2. 上下文窗口没有参与编译

原实现虽然登记 `context_window`，但每个模型拿到的上下文上限相同。

修复后编译器根据模型窗口生成字符预算，预留输出空间，限制每项和总输入大小，并报告
使用量、截断和省略项目。依赖产物优先于普通记忆和历史对话。

### 3. 模型产物默认可被依赖项目读取

这与项目声明的 `explicit_publish` 和人工确认点不一致。

修复后模型与工具产物默认 `private`。跨项目发布必须调用 `publishArtifact()`，且
`approvedBy` 必须为 `user`。

### 4. 来源模型可以被伪造

调用方原本可以把产物标记为任意模型生成。

修复后模型必须存在于注册表，并且确实曾作为该子项目的实际选中模型。未绑定模型的
来源声明会被拒绝。

### 5. 依赖关系缺少循环保护

初版只能在创建时声明已有依赖，尚未提供安全的后续修改接口。

新增 `setDependencies()`，在提交前验证完整依赖图。形成环时拒绝修改并恢复原关系。

### 6. 相同时间戳下的新旧排序不稳定

高频写入或固定测试时钟可能让多条记忆拥有相同时间戳，旧内容可能抢占有限上下文。

修复后每次变更记录 `updated_revision`，上下文优先按项目修订序号排序。

## 结果

- 虚拟模型：5 个。
- 子项目：8 个。
- 实验检查：10 项，全部通过。
- 自动化回归测试：12 项，全部通过。
- 未发布产物、未确认模型记忆和其他子项目原始对话均未泄漏。

## 尚未覆盖

- 真实供应商 tokenizer、费率限制、超时和流式中断。
- 数据库持久化、并发写入和跨设备同步冲突。
- 图片、音频和文件正文传输；当前实验只处理附件元数据。
- 真实模型可能产生的不稳定输出和提示注入，需要主机层沙箱与内容验证。
