# Anna 多模型项目工作区

这个实验把 Anna 的项目管理抽象为三层：

1. **总项目**：保存目标、长期约束、经确认的共享记忆和全部子项目索引。
2. **子项目**：拥有独立对话、任务目标、模型绑定、依赖关系和产物。
3. **上下文包**：Anna 在每次调用模型前按权限和依赖动态编译，不要求不同模型直接
   读取彼此的原始聊天记录。

## 建议界面结构

```text
项目 x                                      [新建子项目]
├─ 项目概览  目标 / 共享记忆 / 关键决策 / 全部产物
├─ 1  OpenAI 推理/代码模型     构建模块一
├─ 2  Gemini 多模态模型        构建模块二
└─ 3  Gemma 本地模型           汇总与测试    依赖：1、2
```

创建子项目时显示四个核心字段：

- 子项目名称与目标；
- 模型选择器，以及模型已经由 Anna 主机确认的能力；
- 需要读取的上游子项目；
- 新产物默认共享范围。

子项目内部仍是熟悉的多模态对话，但侧栏应额外显示“当前模型”“项目记忆”“依赖产物”
和“本次将发送给模型的上下文预览”。用户可以在发送前删除某条记忆或产物。

## 典型流程

总项目 `x` 可以包含：

| 子项目 | 模型绑定示例 | 目标 | 依赖 |
| --- | --- | --- | --- |
| `1` | OpenAI 推理/代码模型 | 构建模块一 | 无 |
| `2` | Gemini 多模态模型 | 构建模块二 | 无 |
| `3` | Gemma 本地模型 | 汇总并测试 | `1`、`2` |

模型名称不是写死在项目格式中的。Anna 主机通过 `registerModel()` 提供当前可用模型及
其已确认能力；如果用户选择的模型不满足任务能力，绑定会回退到 `anna-auto`，并保留
用户原始选择和回退原因。

```js
import { AnnaProjectWorkspace } from "./src/workspace.js";

const workspace = new AnnaProjectWorkspace({
  models: [{
    id: "provider-model-id",
    label: "用户界面中的模型名",
    provider: "provider",
    capabilities: ["text", "code"]
  }]
});

const project = workspace.createProject({
  name: "x",
  goal: "完成多模型协作项目"
});

const one = workspace.createSubproject({
  projectId: project.id,
  name: "1",
  modelId: "provider-model-id",
  requiredCapabilities: ["text", "code"]
});
```

## 记忆共享规则

- 子项目的完整对话默认只进入自己的上下文。
- 用户确认的 `project` 记忆对总项目下所有子项目可见。
- 模型或工具产生的记忆默认是 `proposed`，不会自动成为共享事实。
- `promoteMemory(... approvedBy: "user")` 才会把模型结论提升为已确认项目记忆。
- 模型和工具产物默认是 `private`；用户通过 `publishArtifact()` 才能发布。
- 产物可发布为 `dependencies` 或 `project`。
- 子项目 `3` 声明依赖 `1` 和 `2` 后，可读取二者标记为 `dependencies` 的产物摘要。
- 每条记忆和产物都保存来源子项目、来源模型或工具，便于审计和纠错。

这种设计共享的是**结构化项目知识和产物**，不是把所有聊天记录拼成一个无限增长的
提示词。

## 上下文编译

`compileContext()` 返回一个模型中立的 `anna.context-package`，包含：

- 总项目目标和规则；
- 当前子项目目标与模型绑定；
- 经确认的项目记忆；
- 当前子项目及其显式依赖的产物；
- 当前子项目最近的对话；
- 防止把附件、模型输出或其他子项目内容当成高优先级指令的上下文策略。

编译器会读取所选模型的 `context_window`，预留输出空间后对输入内容实施统一字符预算。
当空间不足时，优先级为：当前任务、项目规则、依赖产物、共享记忆、当前对话。返回值中
的 `context_budget` 会说明实际使用量、截断状态和各类省略数量。字符预算是供应商
tokenizer 接入前的保守估算，真实主机仍应在发送前执行模型自己的 token 计数。

真正调用模型前，Anna 主机仍需重新确认模型能力、上下文窗口、供应商权限和用户是否
同意把相应数据发送给该模型供应商。

## 项目格式

可持久化 JSON 格式定义在
`contracts/project-workspace.schema.json`。核心对象使用 `schema_version: 1`，
便于以后迁移数据库、同步服务或 Anna App UI。

## 验证

```bash
npm run check
npm test
npm run experiment
```

当前版本是核心领域原型，不直接调用任何外部模型，也不上传文件内容。

## 虚拟模型实验

`npm run experiment` 会创建 5 个能力不同的虚拟模型，并执行：

- 代码模型生成接口产物；
- 视觉模型读取图片附件元数据并生成界面分析；
- 512 token 小窗口模型读取前两者的已发布产物并完成整合；
- 纯文本模型触发视觉能力不匹配回退；
- 离线模型触发可用性回退；
- 未发布产物隔离、发布审批、伪造来源和循环依赖故障注入。

完整结果与本轮修复记录见 `EXPERIMENT_REPORT_ZH.md`。

## 推荐实现阶段

1. 当前核心：项目格式、动态模型注册、共享记忆、依赖产物和上下文编译。
2. 本地持久化：SQLite 保存项目事件，文件内容使用内容哈希和对象存储引用。
3. Anna App UI：项目树、模型选择器、依赖选择器、共享审批和上下文预览。
4. 主机调度：把上下文包发送给选定模型，并记录供应商、模型版本、成本和运行结果。
5. 检索与压缩：项目变大后按任务相关度选取记忆，并保留可追溯的摘要版本。
